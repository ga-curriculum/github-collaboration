# QA Pass One

## Copy edit

Want more details on any of the following items? Check out the [Writing Style doc](./writing-style.md)

**Document layout**

- [ ] All documents should start with an `H1` header written in title case (capitalizing all significant words in the header). There should never be another `H1` header in a document.
- [ ] Student-facing documents should have a hero image for the document that immediately follows the opening `H1` header.
- [ ] Microlesson content shouldn't have a table of contents.
- [ ] `H2`-`H4` headers should be written in sentence case.
- [ ] Header levels should not be skipped (an `H3` should not follow an `H1`, for example) and should not directly follow one another (add some written copy or an asset between them).
- [ ] You do sections should all have a header that reads as: 🧠 You do
- [ ] Review question sections should all have a header that reads as: ❓ Review questions

**Text formatting**

- [ ] Variable names are placed in `inline code blocks`.
- [ ] Short lines of code used in copy are placed in `inline code blocks`. Longer lines of code or multi-line blocks of code are placed in code blocks.
- [ ] File names are placed in `inline code blocks`.
- [ ] Keyboard shortcuts are placed in `inline code blocks`. Keyboard shortcuts are provided for macOS, Windows, and Linux.
- [ ] Raw URLs and URIs should be wrapped in `inline code blocks`. URLs should also include a link to that location. Raw URLs are not used without a clear reason and should typically have a text mask.
- [ ] Directories, file names, and paths should be wrapped in `inline code blocks`.
- [ ] Property names should be written as `obj.property` or `property`. Methods should be written as `obj.method()` or `method()`.
- [ ] When instructing students to type text - for example, to fill out a form - use `inline code blocks`.
- [ ] When giving directions for interacting with UI elements, use **bold text** to indicate the content of that element. For example, click the **Play** button.
- [ ] New terms should be identified with *italicized text*. The term should also be *italicized* in the accompanying callout.
- [ ] Adding emphasis is done by using ***bold and italicized text***. This should be done sparingly.

**Modularization and miscellaneous**

- [ ] Any list should have uniform ending punctuation. If any item in a list requires ending punctuation, all items should have ending punctuation.
- [ ] Timeframes are not used in any content, for example, *this afternoon*, *yesterday*, *tomorrow*, etc.
- [ ] Content outside the current module, including labs, should not be mentioned or linked to. You ***can*** reference other microlessons within a module.
- [ ] Any callouts should use the appropriate emojis.
- [ ] No placeholder text (**tktk**) is present.
- [ ] Images should all have alt text.
- [ ] The content is free of grammatical and spelling errors.

## Code review

Want more details on any of the following items? Check out the [Technical Style doc](./technical-style.md).

- [ ] Setup steps have been completed to ensure accuracy (if applicable). Setup should make use of the assumed directory structure students will have.
- [ ] Provided code and commands have been copied out of code blocks and pasted into a project from start to finish to ensure code accuracy.
- [ ] The data used in code is inclusive and clear.
- [ ] Code blocks are all language-labeled and only contain valid code.
- [ ] Prefer `'` in code instead of `"` except when `"` is preferred by the language.
- [ ] Lines of code in code blocks should never exceed 80 characters.
- [ ] Descriptive variable names with appropriate casing are used.
- [ ] Lines end with semicolons in languages that will allow it.
- [ ] To ease copying content, bash and shell code should not include leading indicators such as `$` or `#`.
- [ ] Two-space indentation is used.
